## Citations

En écrivant ['rmw.link](https://rmw.link) ', j'ai senti que j'avais besoin d'une base de données intégrée.

En raison du débit réseau qu'impliquent l'enregistrement, la lecture et l'écriture fréquents, `sqlite3` était trop avancé pour les préoccupations de performance.

Une base de données clé-valeur de niveau inférieur était donc plus appropriée [(lmdb est 10 fois plus rapide que sqlite](https://discourse.world/h/2020/06/05/Shine-and-poverty-key-value-database-LMDB-in-applications-for-iOS) ).

![](https://raw.githubusercontent.com/gcxfd/img/gh-pages/yxZV8x.jpg)

Finalement, j'ai opté pour la version magique de `lmdb` - `mdbx`.

Actuellement, le paquetage existant de `rust` de [mdbx-rs (mdbx-sys)](https://github.com/vorot93/mdbx-rs/issues/1) de `mdbx` [ne supporte pas Windows](https://github.com/vorot93/mdbx-rs/issues/1), donc j'ai pris sur moi d'empaqueter une version avec le support de Windows.

Prise en charge du stockage des types de rouille personnalisés. Prend en charge les accès multithreads.

La base de données peut être définie dans un module à l'aide de `lazy_static`, puis simplement introduite et utilisée avec quelque chose comme :

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
```

## Qu'est-ce que libmdbx ?

[mdbx](https://github.com/erthink/libmdbx) est une base de données secondaire basée sur lmdb, par le russe [Леонид Юрьев (Leonid Yuriev)](https://vk.com/erthink).

[lmdb](https://en.wikipedia.org/wiki/Lightning_Memory-Mapped_Database) est une base de données clé-valeur embarquée super rapide.

Le moteur de recherche en texte intégral [MeiliSearch](https://docs.meilisearch.com/reference/under_the_hood/storage.html#measured-disk-usage) est basé sur lmdb.

[Le cadre d'apprentissage profond caffe utilise également lmdb comme magasin de données](https://docs.nvidia.com/deeplearning/dali/user-guide/docs/examples/general/data_loading/dataloading_lmdb.html).

[mdbx est 30% plus rapide que lmdb dans le test de performance embarqué ioarena](https://github.com/erthink/libmdbx#added-features).

![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-1.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-3.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-4.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-5.png)

Parallèlement, mdbx [améliore de nombreuses lacunes de](https://github.com/erthink/libmdbx#improvements-beyond-lmdb) lmdb. C'est pourquoi Erigon (le client ethereum de nouvelle génération) est récemment passé de LMDB à MDBX [^erigon].

### Exemple 1 : écrire `set(key,val)` et lire `.get(key)`

Prenons un exemple simple, [par exemple/01.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/01.rs).

#### Code

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
use anyhow::{Ok, Result};
use mdbx::prelude::*;

env_rw!(
  MDBX,
  {
    let mut db_path = std::env::current_exe().unwrap();
    db_path.set_extension("mdb");
    println!("mdbx file path {}", db_path.display());
    db_path.into()
  },
  r,
  w
);

mdbx! {
  MDBX // Nom de la variable de la base de données Env
  Test // Test de la base de données
}

fn main() -> Result<()> {
  // Afficher le numéro de version de libmdbx
  unsafe {
    println!(
      "mdbx version https://github.com/erthink/libmdbx/releases/tag/v{}.{}.{}",
      mdbx_version.major, mdbx_version.minor, mdbx_version.release
    );
  }

  // Lecture et écriture multithreads
  let t = std::thread::spawn(|| {
    let tx = w!();
    let test = tx | Test;
    test.set([1, 2], [6])?;
    println!("test1 get {:?}", test.get([1, 2]));

    match test.get([1, 2])? {
      Some(val) => {
        let t: &[u8] = &val;
        println!("{:?}", t);
      }
      None => unreachable!(),
    }
    Ok(())
  });

  t.join().unwrap()?;

  Ok(())
}
```

##### `env_rw!` Définition de la base de données

2. Retourne un objet, [mdbx: : env: : Config](https://docs.rs/mdbx/latest/src/mdbx/env.rs.html#27-35).
  
  Nous utilisons la configuration par défaut, car `Env` implémente `From<Into<PathBuf>>`, donc le chemin de la base de données `into()` fera l'affaire, et la configuration par défaut est la suivante.

##### Fils et transactions

Le code ci-dessus illustre la lecture et l'écriture multithread.

Il est important de noter qu'il **ne** peut y **avoir qu'une seule transaction dans le même thread à un moment donné. Si un thread a plus d'une transaction ouverte, le programme se bloquera**.

La transaction sera engagée à la fin de la portée.

fn main() -> Result<()> {

  w!(Test1.set [2, 3],[4, 5]);

  // Lecture rapide
  match r!(Test1.get [2, 3]) {
    Some(r) => {
      println!(
        "\nu16::from_le_bytes({:?}) = {}",
        r,
        u16::from_le_bytes((*r).try_into()?)
      );
    }
    None => unreachable!(),
  }

  {
    let tx = w!();
    let test1 = tx | Test1;

    test1.set(&[9], &[10, 12])?;
    test1.set([8, 1], [9])?;

      println!("{} = {}", k, v);

    dbg!(test1.del_val([8, 1], [3])?);
    dbg!(test1.get([8, 1])?.unwrap());
    dbg!(test1.del_val([8, 1], [9])?);
    dbg!(test1.get([8, 1])?);

    dbg!(test1.del([9])?);
    dbg!(test1.get([9])?);
    dbg!(test1.del([9])?);

    test2.set(&"abc", &"012")?;

    test3.set(16, 32)?;
    test3.set(-15, 6)?;
    test3.set(-10, 6)?;
    test3.set(-12, 6)?;
    test3.set(0, 6)?;
    test3.set(10, 5)?;

    println!("\n-- loop test3");
    for (k, v) in test3 {
      println!("{:?} = {:?}", k, v);

    test4.set(10, 5)?;

#### Drapeaux de base de données

Vous pouvez voir les drapeaux de base de données ajoutés aux données dans [examples/02.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/02.rs) à `Test4`. `flag DUPSORT`

La base de données libmdbx possède un certain nombre d'indicateurs ( [`MDBX_db_flags_t`](https://erthink.github.io/libmdbx/group__c__dbi.html#gafe3bddb297b3ab0d828a487c5726f76a) ) qui peuvent être définis.

* REVERSEKEY utilise la comparaison inverse des chaînes pour les clés. (utile lorsque vous utilisez des petits numéros codés en bout de ligne comme clés)
* DUPSORT utilise des doublons triés, c'est-à-dire qu'il autorise plusieurs valeurs pour une clé.
* INTEGERKEY Clé numérique native ordonnée par octets uint32_t ou uint64_t. Les clés doivent avoir la même taille et doivent être alignées lorsqu'elles sont passées comme arguments.
* DUPFIXED La taille des valeurs de données doit être la même si DUPSORT est utilisé (permet un comptage rapide du nombre de valeurs).
* DUPSORT et DUPFIXED sont nécessaires pour INTEGERDUP ; les valeurs sont des entiers (similaires à INTEGERKEY). Les valeurs de données doivent toutes avoir la même taille et doivent être alignées lorsqu'elles sont transmises en tant que paramètres.
* REVERSEDUP utilise DUPSORT ; la comparaison inverse des chaînes de caractères est utilisée pour les valeurs de données.
* CREATE crée la BD si elle n'existe pas (ajoutée par défaut).
* DB_ACCEDE Ouvre une sous-base de données existante créée à l'aide de l'indicateur inconnu.  
  Cet indicateur DB_ACCEDE est destiné à ouvrir des sous-bases de données existantes créées avec des indicateurs inconnus (REVERSEKEY, DUPSORT, INTEGERKEY, DUPFIXED, INTEGERDUP et REVERSEDUP).  
  Dans ce cas, la sous-base de données ne renverra pas une erreur INCOMPATIBLE, mais sera ouverte avec les drapeaux utilisés pour la créer, et l'application pourra alors déterminer les drapeaux réels avec mdbx_dbi_flags().

##### DUPSORT : Une clé correspond à plus d'une valeur

`DUPSORT`Cela signifie qu'une clé peut correspondre à plus d'une valeur.

Si vous voulez définir plusieurs drapeaux, écrivez comme suit `flag DUPSORT | DUPFIXED`

##### `.dup(key)` itérateur qui renvoie toutes les valeurs correspondant à une clé

Cette fonction est uniquement disponible pour les bases de données marquées par `DUPSORT` où une clé peut correspondre à plus d'une valeur.

Pour les bases de données `DUPSORT`, `get` ne renvoie que la première valeur pour cette clé. Pour obtenir toutes les valeurs, utilisez `dup`.

##### Drapeaux de base de données ajoutés automatiquement par défaut

Lorsque le type de données est `u32` / `u64` / `usize`, l'indicateur de base de données est automatiquement ajouté. [`INTEGERKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#105) .

Sur les machines dotées d'un système de codage à petite échelle, d'autres types de chiffres sont automatiquement ajoutés. [`REVERSEKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#108) L'indicateur de base de données est automatiquement ajouté lorsque le type de données est / / .

#### Suppression des données

##### `.del(key)` Suppression d'une touche

`.del(val)` Supprime la valeur correspondant à une clé.

Si la base de données a le drapeau `DUPSORT`, toutes les valeurs sous cette clé seront supprimées.

Renvoie `true`si des données sont supprimées, et `false`dans le cas contraire.

##### `.del_val(key,val)` Suppression de la correspondance exacte

`.del_val(key,val)` Supprime les paires clé-valeur qui correspondent exactement aux paramètres d'entrée.

Renvoie `true`si des données sont supprimées, et `false`dans le cas contraire.

#### Traversée

##### traversée séquentielle

En raison de la mise en œuvre de [`std::iter::IntoIterator`](https://doc.rust-lang.org/std/iter/trait.IntoIterator.html) . vous pouvez traverser directement comme suit :

`for (k, v) in test1`

##### `.rev()` Traversée de l'ordre inverse

`for (k, v) in test4.rev()`

##### Triage

Les clés libmdbx sont triées dans l'[ordre du dictionnaire](https://zh.wikipedia.org/wiki/%E5%AD%97%E5%85%B8%E5%BA%8F).

* Pour les nombres non signés
  
  sont triés du plus petit au plus grand car les drapeaux de la base de données sont automatiquement ajoutés ( `u32`/ `u64`/ `usize` sont ajoutés à `INTEGERKEY`, d'autres sont ajoutés à `REVERSEKEY` selon le code machine).
  
* Pour les nombres signés
  
  l'ordre est le suivant : 0 d'abord, puis tous les nombres positifs du plus petit au plus grand, puis tous les nombres négatifs du plus petit au plus grand.

#### `.range(begin..end)` Intervalle Iteration

Pour les nombres, un intervalle est un intervalle numérique.

Pour le binaire, le même intervalle peut être construit, par exemple

```
let begin : &[u8] = &[1,1];
for (k,v) in test0.range(begin..=&[2]) {}
```

Si `begin` est supérieur à `end`, l'itération se fera en sens inverse.

#### `.rev_range` Intervalles inversés

Si vous voulez obtenir un intervalle inversé qui est inférieur ou égal à une valeur, vous pouvez procéder comme suit

```
test2.rev_range(2..)
```

La sortie sera

```
(2, 4)
(1, 5)
(0, 0)
```

L'une des deux options `begin` ou `end` ne doit pas être définie pour l'intervalle inversé, car si les deux sont définies, vous pouvez toujours utiliser `range(end..begin)` pour obtenir le même effet.

#[proc_macro_derive(MdbxSpeedy)]
pub fn mdbx_speedy(ts: TokenStream) -> TokenStream {
  let ast: syn::DeriveInput = syn::parse(ts).unwrap();
  let name = &ast.ident;
  quote! {
    impl mdbx::prelude::FromMdbx for #name {
      fn from_mdbx(_: mdbx::prelude::PtrTx, val: mdbx::prelude::MDBX_val) -> Self {
        Self::read_from_buffer(val_bytes!(val)).unwrap()
      }

      fn to_as_ref(&self) -> Vec<u8> {
        self.write_to_vec().unwrap()
      }
    }

  }

## Notes de bas de page

[^erigon] : [Erigon (le client Ethernet de nouvelle génération) est récemment passé de LMDB à MDBX](https://github.com/ledgerwatch/erigon/wiki/Criteria-for-transitioning-from-Alpha-to-Beta#switch-from-lmdb-to-mdbx).

Ils citent les avantages de la transition de LMDB à MDBX.

> Erigon a commencé avec une base de données BoltDB, puis a ajouté la prise en charge de BadgerDB et a finalement migré complètement vers LMDB. À un moment donné, nous avons rencontré des problèmes de stabilité causés par notre utilisation de LMDB qui n'avaient pas été prévus par les créateurs. Depuis lors, nous avons étudié un dérivé de LMDB bien supporté, appelé MDBX, et nous espérons profiter de ses améliorations en matière de stabilité et éventuellement collaborer davantage à l'avenir. L'intégration de MDBX est maintenant terminée et il est temps de procéder à d'autres tests et à la documentation.
> 
> Avantages de la transition de LMDB à MDBX.
> 
> 1. La croissance "espace (géométrie)" des fichiers de la base de données fonctionne correctement. C'est important, surtout sous Windows. Dans LMDB, il faut spécifier la taille de la carte mémoire une fois à l'avance (actuellement, nous utilisons 2 To par défaut) et si le fichier de la base de données dépasse cette limite, le processus doit être redémarré. Sous Windows, si la taille de la carte mémoire est fixée à 2 To, le fichier de la base de données aura une taille initiale de 2 To, ce qui n'est pas très pratique. Dans MDBX, la taille de la carte mémoire est augmentée par incréments de 2 Go. Cela implique un remappage occasionnel, mais se traduit par une meilleure expérience utilisateur.
>   
> 2. MDBX effectue des contrôles plus stricts sur l'utilisation simultanée du traitement des transactions et le chevauchement des transactions de lecture et d'écriture dans le même fil d'exécution. Cela nous permet de repérer certaines erreurs non évidentes et rend le comportement plus prévisible.  
>   En plus de 5 ans (depuis qu'il a été séparé de LMDB), MDBX a accumulé un grand nombre de correctifs de sécurité et d'heisenbugs qui, à notre connaissance, existent toujours dans LMDB. Certains d'entre eux ont été découverts lors de nos tests, et les responsables de MDBX les ont pris au sérieux et les ont corrigés rapidement.
>   
> 3. Les bases de données qui modifient constamment leurs données créent une grande quantité d'espace récupérable (également appelé "freelist" dans la terminologie LMDB). Nous avons dû patcher LMDB pour corriger les défauts les plus graves dans la gestion de l'espace récupérable ( [analyse)](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide). MDBX a accordé une [attention particulière à la gestion efficace de l'espace récupérable et, jusqu'à présent, n'a pas eu besoin d'être patché](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide%EF%BC%89%E3%80%82MDBX%E5%AF%B9%E5%8F%AF%E5%9B%9E%E6%94%B6%E7%A9%BA%E9%97%B4%E7%9A%84%E6%9C%89%E6%95%88%E5%A4%84%E7%90%86%E8%BF%9B%E8%A1%8C%E4%BA%86%E7%89%B9%E5%88%AB%E7%9A%84%E5%85%B3%E6%B3%A8%EF%BC%8C%E5%88%B0%E7%9B%AE%E5%89%8D%E4%B8%BA%E6%AD%A2%EF%BC%8C%E8%BF%98%E4%B8%8D%E9%9C%80%E8%A6%81%E6%89%93%E8%A1%A5%E4%B8%81%E3%80%82).
>   
> 4. D'après nos tests, MDBX a obtenu des résultats légèrement meilleurs sur nos charges de travail.
>   
> 5. MDBX expose davantage de données de télémétrie interne, c'est-à-dire davantage de mesures sur ce qui se passe dans la base de données. Et nous avons ces données dans Grafana - pour prendre de meilleures décisions sur la conception des applications. Par exemple, après la transition complète vers MDBX (suppression de la prise en charge de LMDB), nous mettrons en œuvre une politique de "commit half full transaction" pour éviter les contacts de disque en surnombre ou non. Cela simplifiera davantage notre code sans affecter les performances.
>   
> 6. MDBX supporte le mode "Ouverture exclusive" - nous l'utilisons pour les migrations de bases de données afin d'empêcher tout autre lecteur d'accéder à la base de données pendant le processus de migration.
