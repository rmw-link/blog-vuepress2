## Citater

Da jeg skrev ["rmw.link](https://rmw.link) ", følte jeg, at jeg havde brug for en indbygget database.

På grund af den netværksgennemstrømning, der er nødvendig for at optage, læse og skrive ofte, var `sqlite3` for avanceret af hensyn til ydeevnen.

Så en nøgle-værdi-database på et lavere niveau var mere passende [(lmdb er 10 gange hurtigere end sqlite](https://discourse.world/h/2020/06/05/Shine-and-poverty-key-value-database-LMDB-in-applications-for-iOS) ).

![](https://raw.githubusercontent.com/gcxfd/img/gh-pages/yxZV8x.jpg)

I sidste ende valgte jeg den magiske version af `lmdb` - `mdbx`.

I øjeblikket [understøtter](https://github.com/vorot93/mdbx-rs/issues/1) den eksisterende `rust` -pakke af [mdbx-rs (mdbx-sys)](https://github.com/vorot93/mdbx-rs/issues/1) fra `mdbx` [ikke windows](https://github.com/vorot93/mdbx-rs/issues/1), så jeg tog det på mig selv at pakke en version med understøttelse af windows.

Understøttelse af lagring af brugerdefinerede rusttyper. Understøtter flertrådet adgang.

Databasen kan defineres i et modul ved hjælp af `lazy_static` og derefter blot introduceres og bruges med noget i stil med :

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
```

## Hvad er libmdbx?

[mdbx](https://github.com/erthink/libmdbx) er en sekundær database baseret på lmdb af den russiske [Леонид Юрьев (Leonid Yuriev)](https://vk.com/erthink).

[lmdb](https://en.wikipedia.org/wiki/Lightning_Memory-Mapped_Database) er en superhurtig indlejret nøgle-værdi-database.

Fuldtekst-søgemaskinen [MeiliSearch](https://docs.meilisearch.com/reference/under_the_hood/storage.html#measured-disk-usage) er baseret på lmdb.

Den [dybe læringsramme caffe bruger også lmdb som datalager](https://docs.nvidia.com/deeplearning/dali/user-guide/docs/examples/general/data_loading/dataloading_lmdb.html).

[mdbx er 30 % hurtigere end lmdb i benchmark-testen for indlejret ydeevne ioarena](https://github.com/erthink/libmdbx#added-features).

![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-1.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-3.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-4.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-5.png)

Samtidig [forbedrer](https://github.com/erthink/libmdbx#improvements-beyond-lmdb) mdbx [mange af](https://github.com/erthink/libmdbx#improvements-beyond-lmdb) lmdb' [s mangler](https://github.com/erthink/libmdbx#improvements-beyond-lmdb), så Erigon (den næste generation af ethereum-klienten) skiftede for nylig fra LMDB til MDBX [^erigon].

### Eksempel 1 : Skrivning `set(key,val)` og læsning `.get(key)`

Lad os se på et enkelt [eksempel/01.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/01.rs)

#### Kode

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
use anyhow::{Ok, Result};
use mdbx::prelude::*;

env_rw!(
  MDBX,
  {
    let mut db_path = std::env::current_exe().unwrap();
    db_path.set_extension("mdb");
    println!("mdbx file path {}", db_path.display());
    db_path.into()
  },
  r,
  w
);

mdbx! {
  MDBX // Variabelnavn for databasen Env
  Test // Test af database
}

fn main() -> Result<()> {
  // Udskriv versionsnummeret for libmdbx
  unsafe {
    println!(
      "mdbx version https://github.com/erthink/libmdbx/releases/tag/v{}.{}.{}",
      mdbx_version.major, mdbx_version.minor, mdbx_version.release
    );
  }

  // Læsning og skrivning med flere tråde
  let t = std::thread::spawn(|| {
    let tx = w!();
    let test = tx | Test;
    test.set([1, 2], [6])?;
    println!("test1 get {:?}", test.get([1, 2]));

    match test.get([1, 2])? {
      Some(val) => {
        let t: &[u8] = &val;
        println!("{:?}", t);
      }
      None => unreachable!(),
    }
    Ok(())
  });

  t.join().unwrap()?;

  Ok(())
}
```

##### Tråde og transaktioner

Ovenstående kode demonstrerer læsning og skrivning med flere tråde.

Det er vigtigt at bemærke, at der **kun** kan **være én transaktion i den samme tråd på et hvilket som helst tidspunkt, hvis en tråd har mere end én transaktion åben, vil programmet gå ned**.

Transaktionen vil blive indgået ved afslutningen af anvendelsesområdet.

    }
    None => unreachable!(),

    test3.set(16, 32)?;
    test3.set(-15, 6)?;
    test3.set(-10, 6)?;
    test3.set(-12, 6)?;
    test3.set(0, 6)?;
    test3.set(10, 5)?;

    println!("\n-- loop test3");

Standarddatatypen er [`Bin`](https://docs.rs/mdbx/latest/mdbx/type/struct.Bin.html) kan der skrives til alle data, der implementerer `AsRef<[u8]>`.

#### Databaseflag

Du kan se de databaseflag, der er tilføjet til dataene i [examples/02.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/02.rs) på `Test4` `flag DUPSORT`

Libmdbx-databasen har en række flag ( [`MDBX_db_flags_t`](https://erthink.github.io/libmdbx/group__c__dbi.html#gafe3bddb297b3ab0d828a487c5726f76a) ), der kan indstilles.

* REVERSEKEY anvender omvendt stringsammenligning for nøgler. (nyttig, når der anvendes små endekodede tal som nøgler)
* DUPSORT anvender sorteret dubletter, dvs. tillader flere værdier for en nøgle.
* INTEGERKEY Native byte-ordered numeriske nøgle uint32_t eller uint64_t. Nøglerne skal have samme størrelse og skal være justeret, når de overføres som et argument.
* DUPFIXED Størrelsen af dataværdierne skal være den samme, hvis DUPSORT anvendes (giver mulighed for en hurtig optælling af antallet af værdier).
* DUPSORT og DUPFIXED er påkrævet for INTEGERDUP; værdierne er hele tal (svarende til INTEGERKEY). Dataværdierne skal alle have samme størrelse og skal være justeret, når de overføres som parametre.
* REVERSEDUP bruger DUPSORT; der anvendes omvendt stringsammenligning for dataværdier.
* CREATE opretter DB'en, hvis den ikke findes (tilføjes som standard).
* DB_ACCEDE Åbner en eksisterende underdatabase, der er oprettet ved hjælp af det ukendte flag.  
  Dette DB_ACCEDE-flag er beregnet til at åbne eksisterende underdatabaser, der er oprettet med ukendte flag (REVERSEKEY, DUPSORT, INTEGERKEY, DUPFIXED, INTEGERDUP og REVERSEDUP).  
  I dette tilfælde returnerer underdatabasen ikke en INCOMPATIBLE-fejl, men åbnes med de flag, der blev brugt til at oprette den, og programmet kan derefter bestemme de faktiske flag med mdbx_dbi_flags().

##### DUPSORT : En nøgle svarer til mere end én værdi

`DUPSORT`, betyder, at en nøgle kan svare til mere end én værdi.

Hvis du ønsker at indstille flere flag, skal du skrive som følger `flag DUPSORT | DUPFIXED`

##### `.dup(key)` iterator, der returnerer alle de værdier, der svarer til en nøgle

Denne funktion er kun tilgængelig for databaser, der er markeret med `DUPSORT`, hvor en nøgle kan svare til mere end én værdi.

For `DUPSORT` -databaser returnerer `get` kun den første værdi for denne nøgle. Hvis du vil have alle værdier, skal du bruge `dup`.

##### Standard automatisk vedhæftede databaseflag

Når datatypen er `u32` / `u64` / `usize`, tilføjes databaseflaget automatisk. [`INTEGERKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#105) .

På maskiner med small-end-kodning tilføjes andre numeriske typer automatisk [`REVERSEKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#108) Databaseflaget tilføjes automatisk, når datatypen er / / / .

#### Sletning af data

##### `.del(key)` Sletning af en nøgle

`.del(val)` Sletter den værdi, der svarer til en nøgle.

Hvis databasen har markeringen `DUPSORT`, slettes alle værdier under denne nøgle.

Returnerer `true`, hvis der er slettet data, og `false`, hvis det ikke er tilfældet.

##### `.del_val(key,val)` Sletning med nøjagtig match

`.del_val(key,val)` Sletter nøgle-værdipar, der svarer nøjagtigt til de indtastede parametre.

Returnerer `true`, hvis der er slettet data, og `false`, hvis det ikke er tilfældet.

##### sekventiel gennemkørsel

På grund af gennemførelsen af [`std::iter::IntoIterator`](https://doc.rust-lang.org/std/iter/trait.IntoIterator.html) . du kan krydse direkte som følger :

`for (k, v) in test1`

##### `.rev()` Gennemløb i omvendt rækkefølge

`for (k, v) in test4.rev()`

##### Sortering

Libmdbx-nøglerne er sorteret i [ordbogsrækkefølge](https://zh.wikipedia.org/wiki/%E5%AD%97%E5%85%B8%E5%BA%8F).

* For tal uden fortegn
  
  er sorteret fra den mindste til den største, fordi databaseflagene automatisk tilføjes ( `u32`/ `u64`/ `usize` tilføjes til `INTEGERKEY`, andre tilføjes til `REVERSEKEY` afhængigt af maskinkoden).
  
* For tal med fortegn
  
  rækkefølgen er: 0 først, derefter alle positive tal fra det mindste til det største, derefter alle negative tal fra det mindste til det største.

#### `.range(begin..end)` Interval Iteration

For tal er et interval et numerisk interval.

For binært kan det samme interval konstrueres, f.eks.

```
let begin : &[u8] = &[1,1];
for (k,v) in test0.range(begin..=&[2]) {}
```

Hvis `begin` er større end `end`, vil den gå baglæns.

Interval-iteration er ikke understøttet [`RangeFull`](https://doc.rust-lang.org/std/ops/struct.RangeFull.html) , dvs. at brugen af `..`ikke understøttes, skal du i stedet bruge den ovenfor nævnte [traversal](#%E9%81%8D%E5%8E%86).

#### `.rev_range` Inverterede intervaller

Hvis du vil have et omvendt interval, der er mindre end eller lig med en værdi, kan du gøre følgende

```
test2.rev_range(2..)
```

Udgangen vil være

```
(2, 4)
(1, 5)
(0, 0)
```

En af `begin` eller `end` må ikke være indstillet for det omvendte interval, for hvis begge er indstillet, kan du altid bruge `range(end..begin)` for at opnå samme effekt.

pub fn mdbx_speedy(ts: TokenStream) -> TokenStream {
  let ast: syn::DeriveInput = syn::parse(ts).unwrap();
  let name = &ast.ident;
  quote! {
    impl mdbx::prelude::FromMdbx for #name {
      fn from_mdbx(_: mdbx::prelude::PtrTx, val: mdbx::prelude::MDBX_val) -> Self {
        Self::read_from_buffer(val_bytes!(val)).unwrap()

        self.write_to_vec().unwrap()
      }

## Fodnoter

[^erigon]: [Erigon (næste generation af Ethernet-klienten) er for nylig skiftet fra LMDB til MDBX](https://github.com/ledgerwatch/erigon/wiki/Criteria-for-transitioning-from-Alpha-to-Beta#switch-from-lmdb-to-mdbx).

De nævner fordelene ved at overgå fra LMDB til MDBX.

> Erigon startede med en BoltDB-database backend, tilføjede derefter understøttelse af BadgerDB og migrerede til sidst helt til LMDB.På et tidspunkt løb vi ind i stabilitetsproblemer, som var forårsaget af vores brug af LMDB, og som ikke var forudset af skaberne. Siden da har vi kigget på et godt understøttet derivat af LMDB kaldet MDBX og håber at kunne bruge deres stabilitetsforbedringer og potentielt samarbejde mere i fremtiden.Integrationen af MDBX er nu færdig, og det er tid til mere testning og dokumentation.
> 
> Fordele ved at gå fra LMDB til MDBX.
> 
> 1. Væksten "rum (geometri)" i databasefilerne fungerer korrekt. Dette er vigtigt, især på Windows. I LMDB skal man angive hukommelseskortstørrelsen én gang på forhånd (i øjeblikket bruger vi 2 TB som standard), og hvis databasefilen vokser ud over denne grænse, skal processen genstartes. På Windows ville det at indstille hukommelseskortstørrelsen til 2 TB gøre databasefilen 2 TB stor til at begynde med, hvilket ikke er særlig praktisk. I MDBX øges hukommelseskortstørrelsen i intervaller på 2 Gb. Det betyder lejlighedsvis omprogrammering, men giver en bedre brugeroplevelse.
>   
> 2. MDBX har strengere kontrol af samtidig brug af transaktionsbehandling og overlappende læse- og skrivetransaktioner i den samme udførelsestråd. Det giver os mulighed for at opdage nogle ikke-oplagte fejl og gør adfærden mere forudsigelig.  
>   I løbet af mere end 5 år (siden det blev adskilt fra LMDB) har MDBX akkumuleret et stort antal sikkerhedsrettelser og heisenbug-rettelser, som, så vidt vi ved, stadig findes i LMDB. Nogle af disse blev opdaget under vores test, og MDBX-vedligeholderne tog dem alvorligt og rettede dem hurtigt.
>   
> 3. Når det drejer sig om databaser, der konstant ændrer data, skaber de en rimelig mængde genbrugsplads (også kendt som "freelist" i LMDB-terminologi). Vi har været nødt til at patche LMDB for at rette de alvorligste mangler i håndteringen af genbrugsplads [(analyse)](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide). MDBX har lagt [særlig vægt på effektiv håndtering af genbrugsplads og har indtil videre ikke haft behov for at blive patchet](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide%EF%BC%89%E3%80%82MDBX%E5%AF%B9%E5%8F%AF%E5%9B%9E%E6%94%B6%E7%A9%BA%E9%97%B4%E7%9A%84%E6%9C%89%E6%95%88%E5%A4%84%E7%90%86%E8%BF%9B%E8%A1%8C%E4%BA%86%E7%89%B9%E5%88%AB%E7%9A%84%E5%85%B3%E6%B3%A8%EF%BC%8C%E5%88%B0%E7%9B%AE%E5%89%8D%E4%B8%BA%E6%AD%A2%EF%BC%8C%E8%BF%98%E4%B8%8D%E9%9C%80%E8%A6%81%E6%89%93%E8%A1%A5%E4%B8%81%E3%80%82).
>   
> 4. Baseret på vores test klarede MDBX sig lidt bedre på vores arbejdsbelastninger.
>   
> 5. MDBX afslører flere interne telemetri-data - flere målinger af, hvad der sker i databasen. Og vi har disse data i Grafana - for at træffe bedre beslutninger om applikationsdesign. For eksempel vil vi efter den fuldstændige overgang til MDBX (hvor vi fjerner understøttelsen af LMDB) implementere en politik for "commit half full transaction" for at undgå overflow/unoverflow diskkontakter. Dette vil forenkle vores kode yderligere uden at påvirke ydeevnen.
>   
> 6. MDBX understøtter "Exclusive open"-tilstand - vi bruger dette til databasemigrationer for at forhindre andre læsere i at få adgang til databasen under migrationsprocessen.
