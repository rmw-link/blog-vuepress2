## Цитаты

Во время написания ['rmw.link](https://rmw.link) ' я почувствовал, что мне нужна встроенная база данных.

Из-за пропускной способности сети, связанной с частой записью, чтением и записью, `sqlite3` был слишком продвинут для решения проблем производительности.

Поэтому более подходящей оказалась база данных с ключами-значениями более низкого уровня [(lmdb в 10 раз быстрее sqlite](https://discourse.world/h/2020/06/05/Shine-and-poverty-key-value-database-LMDB-in-applications-for-iOS) ).

![](https://raw.githubusercontent.com/gcxfd/img/gh-pages/yxZV8x.jpg)

В итоге я выбрал магическую версию `lmdb` - `mdbx`.

В настоящее время существующий `rust` пакет [mdbx-rs (mdbx-sys)](https://github.com/vorot93/mdbx-rs/issues/1) от `mdbx` [не поддерживает windows](https://github.com/vorot93/mdbx-rs/issues/1), поэтому я взял на себя труд упаковать версию с поддержкой windows.

Поддержка хранения пользовательских типов ржавчины. Поддерживает многопоточный доступ.

База данных может быть определена в модуле с помощью `lazy_static`, а затем просто представлена и использована с помощью чего-то вроде :

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
```

## Что такое libmdbx?

[mdbx](https://github.com/erthink/libmdbx) - это вторичная база данных, основанная на lmdb, автором которой является русский [Леонид Юрьев (Леонид Юрьев)](https://vk.com/erthink).

[lmdb](https://en.wikipedia.org/wiki/Lightning_Memory-Mapped_Database) - это сверхбыстрая встроенная база данных с ключевыми значениями.

Полнотекстовая поисковая система [MeiliSearch](https://docs.meilisearch.com/reference/under_the_hood/storage.html#measured-disk-usage) основана на lmdb.

[Система глубокого обучения caffe также использует lmdb в качестве хранилища данных](https://docs.nvidia.com/deeplearning/dali/user-guide/docs/examples/general/data_loading/dataloading_lmdb.html).

[mdbx на 30% быстрее lmdb в тесте встроенной производительности ioarena](https://github.com/erthink/libmdbx#added-features).

![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-1.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-3.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-4.png)  
![](https://raw.githubusercontent.com/wiki/erthink/libmdbx/img/perf-slide-5.png)

В то же время mdbx [улучшает многие недостатки](https://github.com/erthink/libmdbx#improvements-beyond-lmdb) lmdb, поэтому Erigon (клиент ethereum нового поколения) недавно перешел с LMDB на MDBX [^erigon].

### Как запустить пример

Сначала клонируйте кодовую базу `git clone git@github.com:rmw-lib/mdbx.git --depth=1 && cd mdbx`

Затем запустите `cargo run --example 01` и он выполнит `examples/01.rs`

Если это ваш собственный проект, пожалуйста, запустите его сначала:

### Пример 1: Написание `set(key,val)` и чтение `.get(key)`

Давайте рассмотрим простой [пример/01.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/01.rs)

#### Код

```rust
use db::User;

let id = 1234;
let user = r!(User.get id);
use anyhow::{Ok, Result};
use mdbx::prelude::*;

env_rw!(
  MDBX,
  {
    let mut db_path = std::env::current_exe().unwrap();
    db_path.set_extension("mdb");
    println!("mdbx file path {}", db_path.display());
    db_path.into()
  },
  r,
  w
);

mdbx! {
  MDBX // Имя переменной базы данных Env
  Test // Тест базы данных
}

fn main() -> Result<()> {
  // Вывести номер версии libmdbx
  unsafe {
    println!(
      "mdbx version https://github.com/erthink/libmdbx/releases/tag/v{}.{}.{}",
      mdbx_version.major, mdbx_version.minor, mdbx_version.release
    );
  }

  // Многопоточное чтение и запись
  let t = std::thread::spawn(|| {
    let tx = w!();
    let test = tx | Test;
    test.set([1, 2], [6])?;
    println!("test1 get {:?}", test.get([1, 2]));

    match test.get([1, 2])? {
      Some(val) => {
        let t: &[u8] = &val;
        println!("{:?}", t);
      }
      None => unreachable!(),
    }
    Ok(())
  });

  t.join().unwrap()?;

  Ok(())
}
```

##### `env_rw!` Определение базы данных

Код начинается с макроса env_rw, который имеет 4 параметра.

1. Имя переменной среды базы данных
  
2. Возвращает объект, [mdbx:: env:: Config](https://docs.rs/mdbx/latest/src/mdbx/env.rs.html#27-35).
  
  Мы используем конфигурацию по умолчанию, так как `Env` реализует `From<Into<PathBuf>>`, поэтому подойдет путь к базе данных `into()`, а конфигурация по умолчанию выглядит следующим образом.

3. Имя макроса транзакции чтения базы данных, значение по умолчанию - `r`
  
4. Имя макроса транзакции записи в базу данных, значение по умолчанию - `w`

##### Потоки и транзакции

Приведенный выше код демонстрирует многопоточное чтение и запись.

Важно отметить, что **в одном потоке** одновременно может **находиться только одна транзакция, если в потоке открыто более одной транзакции, программа завершится аварийно**.

Сделка будет совершена в конце диапазона.

  MDBX // Имена переменных для базы данных ENV
  Test1
  Test2
    key Str
    val Str
  Test3
    key i32
    val u64
  Test4
    key u64
    val u16
    flag DUPSORT
}

fn main() -> Result<()> {
  // Быстрая запись
  w!(Test1.set [2, 3],[4, 5]);

  // Быстрое чтение
  match r!(Test1.get [2, 3]) {
    Some(r) => {
      println!(
        "\nu16::from_le_bytes({:?}) = {}",
        r,
        u16::from_le_bytes((*r).try_into()?)
      );
    }
    None => unreachable!(),
  }

  // Многочисленные операции над несколькими базами данных в одной транзакции
  {
    let tx = w!();
    let test1 = tx | Test1;

    test1.set(&[9], &[10, 12])?;
    test1.set([8, 1], [9])?;
    test1.set("rmw.link", "Down with Data Hegemony")?;
    test1.set(&"abc", &"012")?;

    println!("\n-- loop test1");
    for (k, v) in test1 {
      println!("{} = {}", k, v);
    }

    dbg!(test1.del_val([8, 1], [3])?);
    dbg!(test1.get([8, 1])?.unwrap());
    dbg!(test1.del_val([8, 1], [9])?);
    dbg!(test1.get([8, 1])?);

    dbg!(test1.del([9])?);
    dbg!(test1.get([9])?);
    dbg!(test1.del([9])?);

    let test2 = tx | Test2;
    test2.set("rmw.link", "Down with Data Hegemony")?;
    test2.set(&"abc", &"012")?;
    println!("\n-- loop test2");
    for (k, v) in test2 {
      println!("{} = {}", k, v);
    }

    let test3 = tx | Test3;

    test3.set(13, 32)?;
    test3.set(16, 32)?;
    test3.set(-15, 6)?;
    test3.set(-10, 6)?;
    test3.set(-12, 6)?;
    test3.set(0, 6)?;
    test3.set(10, 5)?;

    println!("\n-- loop test3");
    for (k, v) in test3 {
      println!("{:?} = {:?}", k, v);

#### Флаги базы данных

Вы можете увидеть флаги базы данных, добавленные к данным в файле [examples/02.rs](https://github.com/rmw-lib/mdbx/blob/master/examples/02.rs) по адресу `Test4`. `flag DUPSORT`

База данных libmdbx имеет ряд флагов ( [`MDBX_db_flags_t`](https://erthink.github.io/libmdbx/group__c__dbi.html#gafe3bddb297b3ab0d828a487c5726f76a) ), которые могут быть установлены.

* REVERSEKEY использует обратное сравнение строк для ключей. (полезно при использовании небольших конечных кодовых чисел в качестве ключей)
* DUPSORT использует отсортированные дубликаты, т.е. допускает несколько значений для ключа.
* INTEGERKEY Родной упорядоченный по байтам числовой ключ uint32_t или uint64_t. Ключи должны быть одного размера и должны быть выровнены при передаче в качестве аргументов.
* DUPFIXED Размер значений данных должен быть одинаковым, если используется DUPSORT (позволяет быстро подсчитать количество значений).
* DUPSORT и DUPFIXED требуются для INTEGERDUP; значения являются целыми числами (аналогично INTEGERKEY). Все значения данных должны иметь одинаковый размер и должны быть выровнены при передаче в качестве параметров.
* REVERSEDUP использует DUPSORT; для значений данных используется обратное сравнение строк.
* CREATE создает БД, если она не существует (добавляется по умолчанию).
* DB_ACCEDE Открывает существующую подбазу данных, созданную с использованием флага неизвестности.  
  Этот флаг DB_ACCEDE предназначен для открытия существующих подбаз данных, созданных с неизвестными флагами (REVERSEKEY, DUPSORT, INTEGERKEY, DUPFIXED, INTEGERDUP и REVERSEDUP).  
  В этом случае подбаза данных не вернет ошибку INCOMPATIBLE, но будет открыта с флагами, использованными для ее создания, и приложение сможет определить фактические флаги с помощью mdbx_dbi_flags().

##### DUPSORT : Один ключ соответствует более чем одному значению

`DUPSORT`Это означает, что ключ может соответствовать более чем одному значению.

Если вы хотите установить несколько флагов, напишите следующим образом `flag DUPSORT | DUPFIXED`

##### `.dup(key)` итератор, возвращающий все значения, соответствующие ключу

Эта функция доступна только для баз данных, помеченных `DUPSORT`, где ключ может соответствовать более чем одному значению.

Для баз данных `DUPSORT`, `get` возвращает только первое значение для этого ключа. Чтобы получить все значения, используйте `dup`.

##### Флаги базы данных, автоматически добавляемые по умолчанию

Если тип данных `u32` / `u64` / `usize`, флаг базы данных добавляется автоматически. [`INTEGERKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#105) .

На машинах с кодировкой малого конца автоматически добавляются другие типы цифр [`REVERSEKEY`](https://docs.rs/mdbx-proc/latest/src/mdbx_proc/lib.rs.html#108) Флаг базы данных автоматически добавляется, если тип данных - / / .

#### Удаление данных

##### `.del(key)` Удаление клавиши

`.del(val)` Удаляет значение, соответствующее ключу.

Если база данных имеет флаг `DUPSORT`, все значения под этим ключом будут удалены.

Возвращает `true`, если данные удалены, и `false`, если нет.

##### `.del_val(key,val)` Удаление точного совпадения

`.del_val(key,val)` Удаляет пары ключ-значение, которые точно соответствуют входным параметрам.

Возвращает `true`, если данные удалены, и `false`, если нет.

#### Обход

##### последовательный обход

В связи с реализацией [`std::iter::IntoIterator`](https://doc.rust-lang.org/std/iter/trait.IntoIterator.html) . вы можете пройти напрямую следующим образом:

`for (k, v) in test1`

##### `.rev()` Обход обратного порядка

`for (k, v) in test4.rev()`

##### Сортировка

Ключи libmdbx сортируются в [порядке словаря](https://zh.wikipedia.org/wiki/%E5%AD%97%E5%85%B8%E5%BA%8F).

* Для беззнаковых чисел
  
  сортируются от наименьшего к наибольшему, поскольку флаги базы данных добавляются автоматически ( `u32`/ `u64`/ `usize` добавляются к `INTEGERKEY`, другие добавляются к `REVERSEKEY` в зависимости от машинного кода).
  
* Для знаковых чисел
  
  порядок следующий: сначала 0, затем все положительные числа от наименьшего до наибольшего, затем все отрицательные числа от наименьшего до наибольшего.

#### `.range(begin..end)` Интервал Итерация

Для чисел интервал - это числовой интервал.

Для двоичных чисел можно построить такой же интервал, напр.

```
let begin : &[u8] = &[1,1];
for (k,v) in test0.range(begin..=&[2]) {}
```

Если `begin` больше, чем `end`, итерация выполняется в обратном порядке.

Например, `test1.range(5..2)` выведет следующее :

#### `.rev_range` Инвертированные интервалы

Если вы хотите получить инвертированный интервал, который меньше или равен значению, вы можете сделать следующее

```
test2.rev_range(2..)
```

На выходе получится

```
(2, 4)
(1, 5)
(0, 0)
```

Один из `begin` или `end` не должен быть установлен для инвертированного интервала; потому что если оба установлены, вы всегда можете использовать `range(end..begin)` для достижения того же эффекта.

#[proc_macro_derive(MdbxSpeedy)]
pub fn mdbx_speedy(ts: TokenStream) -> TokenStream {
  let ast: syn::DeriveInput = syn::parse(ts).unwrap();
  let name = &ast.ident;
  quote! {
    impl mdbx::prelude::FromMdbx for #name {
      fn from_mdbx(_: mdbx::prelude::PtrTx, val: mdbx::prelude::MDBX_val) -> Self {
        Self::read_from_buffer(val_bytes!(val)).unwrap()

      }
    }

  }
  .into()

## Сноски

[^erigon]: [Erigon (клиент Ethernet нового поколения) недавно перешел с LMDB на MDBX](https://github.com/ledgerwatch/erigon/wiki/Criteria-for-transitioning-from-Alpha-to-Beta#switch-from-lmdb-to-mdbx).

Они приводят преимущества перехода с LMDB на MDBX.

> Erigon начинал с бэкенда базы данных BoltDB, затем добавил поддержку BadgerDB, и, наконец, полностью перешел на LMDB. В какой-то момент мы столкнулись с проблемами стабильности, вызванными использованием LMDB, которые не были предусмотрены создателями. С тех пор мы смотрим на хорошо поддерживаемую производную LMDB под названием MDBX и надеемся использовать их улучшения стабильности и потенциально сотрудничать с ними в будущем. Интеграция MDBX теперь завершена, и пришло время для дальнейшего тестирования и документирования.
> 
> Преимущества перехода с LMDB на MDBX.
> 
> 1. Рост "пространства (геометрии)" файлов базы данных работает правильно. Это важно, особенно в Windows. В LMDB необходимо один раз заранее указать размер карты памяти (в настоящее время по умолчанию мы используем 2 Тб), и если файл базы данных вырастет сверх этого предела, процесс необходимо перезапустить. В Windows установка размера карты памяти в 2 Тб приведет к тому, что файл базы данных изначально будет иметь размер 2 Тб, что не очень удобно. В MDBX размер карты памяти увеличивается с шагом в 2 Гб. Это означает периодическое переключение, но в результате повышается удобство использования.
>   
> 2. MDBX имеет более строгие проверки на одновременное использование обработки транзакций и перекрытие транзакций чтения и записи в одном потоке выполнения. Это позволяет нам выявить некоторые неочевидные ошибки и делает поведение более предсказуемым.  
>   За более чем 5 лет (с момента отделения от LMDB) MDBX накопил большое количество исправлений безопасности и heisenbug, которые, насколько нам известно, все еще существуют в LMDB. Некоторые из них были обнаружены во время нашего тестирования, и сопровождающие MDBX отнеслись к ним серьезно и быстро исправили их.
>   
> 3. Когда речь идет о базах данных, которые постоянно модифицируют данные, они создают достаточное количество восстанавливаемого пространства (также известного как "freelist" в терминологии LMDB). Нам пришлось внести исправления в LMDB, чтобы устранить наиболее серьезные недостатки в работе с возвращаемым пространством [(анализ)](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide). MDBX уделил [особое внимание эффективной работе с возвращаемым пространством и до сих пор не нуждался в исправлениях](https://github.com/ledgerwatch/erigon/wiki/LMDB-freelist-illustrated-guide%EF%BC%89%E3%80%82MDBX%E5%AF%B9%E5%8F%AF%E5%9B%9E%E6%94%B6%E7%A9%BA%E9%97%B4%E7%9A%84%E6%9C%89%E6%95%88%E5%A4%84%E7%90%86%E8%BF%9B%E8%A1%8C%E4%BA%86%E7%89%B9%E5%88%AB%E7%9A%84%E5%85%B3%E6%B3%A8%EF%BC%8C%E5%88%B0%E7%9B%AE%E5%89%8D%E4%B8%BA%E6%AD%A2%EF%BC%8C%E8%BF%98%E4%B8%8D%E9%9C%80%E8%A6%81%E6%89%93%E8%A1%A5%E4%B8%81%E3%80%82).
>   
> 4. По результатам нашего тестирования, MDBX показал себя немного лучше на наших рабочих нагрузках.
>   
> 5. MDBX раскрывает больше данных внутренней телеметрии - больше метрик о том, что происходит внутри базы данных. И у нас есть эти данные в Grafana - для принятия лучших решений по проектированию приложений. Например, после полного перехода на MDBX (удаление поддержки LMDB), мы внедрим политику "фиксации наполовину полной транзакции", чтобы избежать переполнения/недополнения дисковых контактов. Это еще больше упростит наш код без ущерба для производительности.
>   
> 6. MDBX поддерживает режим "Exclusive open" - мы используем его для миграции баз данных, чтобы предотвратить доступ к базе данных других читателей во время процесса миграции.
